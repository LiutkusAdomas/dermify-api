## Directory Purposes

**`cmd/`:**
- Purpose: CLI command definitions using Cobra
- Contains: Root command, `serve` command (starts server), `version` command (prints commit)
- Key files: `cmd/root.go` (entry point for CLI), `cmd/serve.go` (creates and starts `api.App`)

**`config/`:**
- Purpose: Application configuration structs and Viper-based loading
- Contains: Go structs mapping to `config.yaml`, `Configure()` function, tests
- Key files: `config/config.go` (all config structs and loader)

**`docs/`:**
- Purpose: Auto-generated Swagger/OpenAPI documentation
- Contains: Generated `docs.go` from swag annotations
- Key files: `docs/docs.go` (generated -- do not edit manually)
- Generated: Yes (by `make swagger` / `swag init`)
- Committed: Yes

**`internal/api/`:**
- Purpose: Core API application wiring -- App struct, route delegation, middleware application
- Contains: `App` struct with constructor and server lifecycle
- Key files: `internal/api/api.go` (App struct), `internal/api/routes.go` (route delegation), `internal/api/middleware.go` (metrics middleware bridge)

**`internal/api/apierrors/`:**
- Purpose: Structured JSON error responses with machine-readable codes
- Contains: `ErrorResponse` struct, error code string constants, `WriteError()` utility
- Key files: `internal/api/apierrors/apierrors.go`

**`internal/api/auth/`:**
- Purpose: Authentication utilities -- JWT, password hashing, refresh token operations
- Contains: Claims struct, token generation/validation, bcrypt helpers, refresh token SQL operations
- Key files: `internal/api/auth/auth.go`

**`internal/api/handlers/`:**
- Purpose: HTTP handler functions organized by domain
- Contains: One file per domain (auth, login, users, health, hello, foo, prometheus), shared models
- Key files: `internal/api/handlers/auth.go`, `internal/api/handlers/login.go`, `internal/api/handlers/users.go`, `internal/api/handlers/models.go`

**`internal/api/metrics/`:**
- Purpose: Prometheus metric definitions and Client abstraction
- Contains: Client struct with registry, counter factories, named increment methods
- Key files: `internal/api/metrics/prometheus.go` (Client), `internal/api/metrics/metrics.go` (counter definitions)

**`internal/api/middleware/`:**
- Purpose: HTTP middleware functions for cross-cutting concerns
- Contains: CORS, logging, request ID, Prometheus metrics, JWT authentication middleware
- Key files: `internal/api/middleware/auth.go` (JWT), `internal/api/middleware/cors_config.go` (config CORS), `internal/api/middleware/metrics.go` (Prometheus)

**`internal/api/routes/`:**
- Purpose: Modular route registration using Manager pattern
- Contains: Manager struct orchestrating domain route groups
- Key files: `internal/api/routes/manager.go` (Manager), `internal/api/routes/auth.go`, `internal/api/routes/users.go`, `internal/api/routes/api.go`

**`internal/pkg/`:**
- Purpose: Shared internal utilities (database connection, migrations)
- Contains: PostgreSQL connection helper with retry, Goose migration runner
- Key files: `internal/pkg/postgres.go`

**`migrations/`:**
- Purpose: Goose SQL migration files embedded into the binary
- Contains: SQL files with `-- +goose Up` / `-- +goose Down` sections, embed FS declaration
- Key files: `migrations/fs.go` (embed directive), SQL files

## Key File Locations

**Entry Points:**
- `main.go`: Binary entrypoint, calls `cmd.Execute()`
- `cmd/serve.go`: Creates `api.App` and calls `Start()`
- `internal/api/api.go`: `App.Start()` -- DB connection, migrations, router setup, HTTP server

**Configuration:**
- `config.yaml`: Default config values (port, database, auth, cors)
- `config/config.go`: Config structs and Viper loader
- `golangci.yaml`: Linter configuration
- `Makefile`: Build and development targets
- `Dockerfile`: Multi-stage Docker build
- `docker-compose.yml`: PostgreSQL + API Docker services

**Core Logic:**
- `internal/api/handlers/auth.go`: Registration, logout, token refresh, profile handlers
- `internal/api/handlers/login.go`: Login handler with JWT + refresh token generation
- `internal/api/auth/auth.go`: JWT generation/validation, password hashing, refresh token CRUD
- `internal/api/apierrors/apierrors.go`: Error codes and structured error writing

**Route Registration:**
- `internal/api/routes/manager.go`: Central route manager wiring all domain routes
- `internal/api/routes/auth.go`: Auth route group with protected sub-group
- `internal/api/routes/users.go`: User CRUD route group
- `internal/api/routes/api.go`: General API routes (health, hello, foo)

**Middleware:**
- `internal/api/middleware/auth.go`: JWT Bearer token validation
- `internal/api/middleware/cors_config.go`: Config-driven CORS (used in production)
- `internal/api/middleware/logging.go`: Structured request logging
- `internal/api/middleware/metrics.go`: Prometheus per-request metrics
- `internal/api/middleware/request_id.go`: UUID request ID generation

**Database:**
- `internal/pkg/postgres.go`: Connection with retry and migration runner
- `migrations/20251213115406_users_table.sql`: Users table schema
- `migrations/20260226120000_refresh_tokens_table.sql`: Refresh tokens table schema

**Testing:**
- `config/config_test.go`: Config loading tests (only test file in the project)

## Naming Conventions

**Files:**
- Snake_case for all Go files: `api.go`, `request_id.go`, `cors_config.go`
- One file per domain in handlers: `auth.go`, `login.go`, `users.go`, `health.go`
- `doc.go` in every package for package-level documentation
- `models.go` for shared response structs within handlers
- Migration files: `YYYYMMDDHHMMSS_description.sql`

**Directories:**
- Lowercase, short names: `cmd/`, `config/`, `internal/`, `migrations/`
- Domain grouping under `internal/api/`: `handlers/`, `middleware/`, `routes/`, `metrics/`, `auth/`, `apierrors/`

**Functions:**
- Handlers: `Handle{Action}()` -- e.g., `HandleLogin()`, `HandleRegister()`, `HandleGetProfile()`
- Middleware: Descriptive function names returning `func(http.Handler) http.Handler` -- e.g., `RequireAuth()`, `RequestID`, `NewLoggingMiddleware()`
- Constructors: `New()` or `New{Type}()` -- e.g., `New()` for metrics Client, `NewManager()`, `NewAuthRoutes()`
- Config loader: `Configure(filepath)`

**Types:**
- PascalCase for exported types: `App`, `Configuration`, `Client`, `Manager`, `Claims`
- camelCase for unexported request/response types: `loginRequest`, `registerResponse`, `fooResponse`
- Exported response models in `models.go`: `MessageResponse`, `HealthResponse`, `UserResponse`
- Context key types: unexported empty structs (`requestIDKey struct{}`, `userClaimsKey struct{}`)

## Where to Add New Code

**New Domain (e.g., products, appointments):**
1. Handler file: `internal/api/handlers/{domain}.go` -- define request/response types and `Handle{Action}()` functions
2. Route file: `internal/api/routes/{domain}.go` -- create `{Domain}Routes` struct with `RegisterRoutes(chi.Router)` method
3. Register in Manager: add field to `Manager` struct in `internal/api/routes/manager.go`, instantiate in `NewManager()`, call `RegisterRoutes()` in `RegisterAllRoutes()`
4. Migration: `migrations/YYYYMMDDHHMMSS_{description}.sql` with `-- +goose Up` and `-- +goose Down` sections
5. Response models: add shared response types to `internal/api/handlers/models.go` or keep domain-specific types in the handler file

**New Middleware:**
- Create file: `internal/api/middleware/{name}.go`
- Pattern: function returning `func(http.Handler) http.Handler`
- Global middleware: add `r.Use()` call in `internal/api/api.go` `Start()` method
- Per-route middleware: add `r.Use()` inside a `r.Group()` in the relevant route file

**New Metric:**
1. Define counter factory in `internal/api/metrics/metrics.go`
2. Register in `New()` and add increment method on `Client` in `internal/api/metrics/prometheus.go`
3. Call increment method from handlers

**New Error Code:**
- Add constant to appropriate group in `internal/api/apierrors/apierrors.go`
- Use via `apierrors.WriteError(w, statusCode, apierrors.NewCode, "message")`

**New CLI Command:**
- Create file: `cmd/{command}.go`
- Define `new{Command}Cmd()` returning `*cobra.Command`
- Register in `cmd/root.go` `Execute()` function via `rootCmd.AddCommand()`

**New Utility:**
- Shared internal utility: `internal/pkg/{name}.go`
- Package-specific helper: within the relevant package under `internal/api/`

**New Migration:**
- File: `migrations/YYYYMMDDHHMMSS_{description}.sql`
- Must include both `-- +goose Up` and `-- +goose Down` sections
- Embedded automatically via `//go:embed *.sql` in `migrations/fs.go`
- Runs automatically on server start

## Special Directories

**`internal/`:**
- Purpose: All application code -- Go compiler prevents external packages from importing
- Generated: No
- Committed: Yes

**`docs/`:**
- Purpose: Swagger documentation generated from handler annotations
- Generated: Yes (by `make swagger` / `swag init -g internal/api/api.go --parseDependency --parseInternal -o docs`)
- Committed: Yes
- Regenerate after adding or modifying handler swagger annotations

**`migrations/`:**
- Purpose: Goose SQL migration files, embedded into binary at compile time
- Generated: No (manually authored)
- Committed: Yes
- Files are embedded via `//go:embed *.sql` in `migrations/fs.go`

**`cmd/server/` and `cmd/server/http/`:**
- Purpose: Legacy empty directories (unused)
- Generated: No
- Committed: Yes (but contain no files)

**`dermify/`:**
- Purpose: Empty directory (unused)
- Generated: No
- Committed: Yes (but contains no files)

**`config/` (YAML files at root):**
- Purpose: Additional config YAML files with numeric names (e.g., `config/1290504905.yaml`)
- Status: Untracked -- appear to be test or generated configs
