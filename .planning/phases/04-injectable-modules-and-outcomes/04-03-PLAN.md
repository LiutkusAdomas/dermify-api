    **Filler handler (internal/api/handlers/filler_module.go):**
    Create 3 handler functions following the exact pattern of ipl_module.go:
    - `HandleCreateFillerModule(svc *service.InjectableModuleService, m *metrics.Client) http.HandlerFunc`:
      - Get claims via middleware.GetClaims
      - Parse session ID from chi.URLParam "id"
      - Decode request body into createFillerModuleRequest struct (ProductID int64, plus all optional clinical fields matching domain type)
      - Build domain.FillerModuleDetail from request
      - Call svc.CreateFillerModule(ctx, sessionID, detail, claims.UserID)
      - On error: handleInjectableModuleError
      - On success: m.IncrementInjectableModuleCreatedCount(), write 201 with toFillerModuleDetailResponse
    - `HandleGetFillerModule(svc *service.InjectableModuleService, m *metrics.Client) http.HandlerFunc`:
      - Parse moduleId from chi.URLParam "moduleId"
      - Call svc.GetFillerModule(ctx, moduleID)
      - On error: handleInjectableModuleError
      - On success: write 200 with toFillerModuleDetailResponse
    - `HandleUpdateFillerModule(svc *service.InjectableModuleService, m *metrics.Client) http.HandlerFunc`:
      - Parse moduleId, decode request body into updateFillerModuleRequest
      - Build domain.FillerModuleDetail, set ID from URL
      - Call svc.UpdateFillerModule(ctx, detail, claims.UserID)
      - On error: handleInjectableModuleError
      - On success: re-fetch via svc.GetFillerModule, write 200 (consistent with energy module re-fetch pattern)

    **Botulinum handler (internal/api/handlers/botulinum_module.go):**
    Same 3 handler pattern as filler. Key difference:
    - createBotulinumModuleRequest includes InjectionSites json.RawMessage field
    - The request struct maps directly to domain.BotulinumModuleDetail including reconstitution fields
    - Same create/get/update flow using svc.CreateBotulinumModule/GetBotulinumModule/UpdateBotulinumModule

    **Outcome handler (internal/api/handlers/outcome.go):**
    Create 3 handler functions following the consent handler pattern (singleton per session):
    - `HandleRecordOutcome(svc *service.OutcomeService, m *metrics.Client) http.HandlerFunc`:
      - Parse session ID from chi.URLParam "id"
      - Decode into recordOutcomeRequest (OutcomeStatus string required, EndpointIDs []int64 optional, AftercareNotes/RedFlagsText/ContactInfo/FollowUpAt/Notes optional)
      - Build domain.SessionOutcome, set SessionID, CreatedBy, UpdatedBy from claims
      - Call svc.RecordOutcome(ctx, outcome)
      - On error: map outcome errors (ErrOutcomeAlreadyExists->409, ErrInvalidOutcomeData->400, ErrSessionNotReady->409, ErrOutcomeNotFound->404, default->500)
      - On success: m.IncrementOutcomeRecordedCount(), write 201
    - `HandleGetOutcome(svc *service.OutcomeService, m *metrics.Client) http.HandlerFunc`:
      - Parse session ID, call svc.GetBySessionID, write 200 with toSessionOutcomeResponse
    - `HandleUpdateOutcome(svc *service.OutcomeService, m *metrics.Client) http.HandlerFunc`:
      - Parse session ID, decode request, build domain struct, call svc.UpdateOutcome
      - On success: re-fetch via svc.GetBySessionID, write 200
    Create a local `handleOutcomeError(w, err)` function in outcome.go for outcome-specific error mapping.
