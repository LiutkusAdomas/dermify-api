**4. Create 4 handler files** -- one per module type, following per-file-per-domain convention:

Each handler file (e.g., `ipl_module.go`) contains:

a. **Request types:**
   - `createIPLModuleRequest` with all clinical parameter fields (matching the domain type) plus `DeviceID int64` and `HandpieceID *int64`. All clinical fields are pointers.
   - `updateIPLModuleRequest` with same fields plus `Version int`.

b. **HandleCreateIPLModule(svc *service.EnergyModuleService, m *metrics.Client) func(http.ResponseWriter, *http.Request):**
   - Get claims from context (401 if nil)
   - Parse session ID from URL via `parseIDParam(r)` -- use the existing helper which reads chi.URLParam(r, "id")
   - Decode request body
   - Map request to `domain.IPLModuleDetail` (set DeviceID, HandpieceID, all clinical fields)
   - Call `svc.CreateIPLModule(ctx, sessionID, &detail, claims.UserID)`
   - On error: call `handleEnergyModuleError(w, err)` (shared error handler, defined below)
   - On success: `m.IncrementEnergyModuleCreatedCount()`, write 201 with response

c. **HandleGetIPLModule(svc *service.EnergyModuleService, m *metrics.Client) func(http.ResponseWriter, *http.Request):**
   - Parse session ID and module ID from URL (moduleId via `chi.URLParam(r, "moduleId")` + strconv.ParseInt)
   - Call `svc.GetIPLModule(ctx, moduleID)` -- note: sessionID parsed for URL context but the service method only needs moduleID
   - On error: handle, on success: write 200 with response

d. **HandleUpdateIPLModule(svc *service.EnergyModuleService, m *metrics.Client) func(http.ResponseWriter, *http.Request):**
   - Get claims, parse session ID and module ID
   - Decode request body (includes Version for optimistic locking)
   - Build domain.IPLModuleDetail with ID set from the parsed moduleId, Version from request
   - Call `svc.UpdateIPLModule(ctx, &detail, claims.UserID)`
   - On error: handle, on success: re-fetch via GetIPLModule and write 200 with response

e. **Shared error handler** -- create ONE `handleEnergyModuleError` function (put it in whichever handler file is created first, e.g., ipl_module.go, or create a small `energy_module_errors.go`). It maps:
   - ErrConsentRequired -> 422, ConsentRequired
   - ErrSessionNotEditable -> 409, SessionNotEditable
   - ErrSessionNotFound -> 404, SessionNotFound
   - ErrDeviceNotFound -> 404, RegistryDeviceNotFound
   - ErrDeviceTypeMismatch -> 400, ModuleDeviceTypeMismatch
   - ErrHandpieceMismatch -> 400, ModuleHandpieceMismatch
   - ErrModuleDetailNotFound -> 404, ModuleDetailNotFound
   - ErrModuleDetailVersionConflict -> 409, ModuleDetailVersionConflict
   - ErrInvalidModuleData -> 400, ModuleInvalidData
   - ErrInvalidSessionData -> 400, ModuleInvalidData
   - default -> 500, ModuleCreationFailed

f. **Converter functions:** `toIPLModuleDetailResponse(d *domain.IPLModuleDetail) IPLModuleDetailResponse` etc.

The NdYAG, CO2, and RF handler files follow the exact same structure as IPL -- just with their respective field sets and service method names. All 4 files use the SAME shared `handleEnergyModuleError` function.
